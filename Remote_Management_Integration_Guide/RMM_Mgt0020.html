Managing WSDL files for your web service calls

The Endpoint Protection Web Service Description Language (WSDL) files provide the information that you need to create your web service client and configure it to complete a web service call successfully. These files can be generated dynamically on an installed instance of Symantec Endpoint Protection Manager, or you can manually edit the static WSDL files that are included in the SDK.

Part of a sample WSDL file looks like the following:

<service name="ClientService">    
    <port name="ClientServicePort" binding="tns:ClientServicePortBinding">      
        <soap:address location="https://<ihostname>:port_number/sepm/ws/v1/ClientService"/>;    
    </port>  
</service>

where hostname and port_number are the hostname or IP address and port number of the Symantec Endpoint Protection Manager with which a web service client communicates. These values are provided if the files are generated dynamically on the management server; they are manually edited if the web service client uses the static WSDL files.

The Endpoint Protection WSDL types element is specified in a referenced XML schema (.xsd). To call a specific web service, make sure that you have both the WSDL file and its referenced XSD file. You need a WSDL file and its paired XSD for each of the following five web services:

  • LicenseService

  • ClientService

  • CommandService

  • LiveUpdateService

  • PolicyService

If you develop a remote management application using a higher-level language such as Java or any of the .NET languages, you may find it easier to edit the static files. Otherwise, your application must provide a mechanism for passing the WSDL files back from the Symantec Endpoint Protection Manager server to your web service clients. See Static WSDL files.

If you are a system administrator writing PowerShell scripts against your own management server, you may find it more useful to use the dynamically generated files. See Dynamic WSDL files.
